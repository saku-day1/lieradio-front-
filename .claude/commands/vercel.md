新しい `api/*.js` の追加・既存の変更、または `vercel.json` の設定変更を対象に、以下の7項目を順番にチェックしてください。

各項目は「✅ 問題なし」「⚠️ 要注意」「❌ 設計違反」の3段階で評価し、問題がある場合は該当箇所と修正案を具体的に示すこと。

---

## 1. Serverless 前提

Vercel Serverless Functions の制約を前提とした実装になっているか確認する。

チェック観点：
- 1リクエスト = 1関数呼び出しの前提で設計されているか
- 関数の実行時間が長くなりすぎていないか（Vercel の最大実行時間: Hobby プランは 10 秒、Pro は 60 秒）
- ファイルシステムへの書き込み（`fs.writeFile` 等）をリクエスト間で共有する前提になっていないか（`/tmp` は同一インスタンス内のみ有効）
- 関数の起動ごとに初期化コストが発生する処理（DB 接続・大量データのパース等）をキャッシュ戦略なしで毎回実行していないか

## 2. 永続メモリ不可

インメモリ状態（モジュールスコープの変数・Map・配列等）をリクエスト間で共有する前提になっていないか確認する。

チェック観点：
- モジュールスコープに `let cache = {}` のような変数を持ち、リクエスト間で使い回す設計になっていないか（Vercel は複数インスタンスを起動するためインメモリ状態は無効）
- レートリミット・カウンター・セッション管理をインメモリで実装していないか（Redis で持つこと）
- 「一度読んだら再利用」を前提としたシングルトンパターンがないか
- インメモリキャッシュが「同一インスタンム内のウォームアップ効果」として許容範囲内か、「クロスリクエストで必ず共有される」誤解で書かれていないか

## 3. Cron 運用

`vercel.json` の Cron 設定、または Cron から呼ばれるエンドポイントの変更を確認する。

チェック観点：
- Cron エンドポイント（`episodes-refresh.js`・`episode-meta.js` 等）に `CRON_SECRET` による認証チェックがあるか
- 認証なしで外部から叩けるエンドポイントになっていないか
- Cron の実行間隔が API クォータ（YouTube API・Sheets API）に対して適切か
- Cron が失敗したときのフォールバック（`data/episodes.json` 等）が機能するか
- 新しい Cron を追加する場合、既存のものと実行タイミングが競合しないか

## 4. Edge Runtime 考慮

Edge Runtime で動作させる場合と、Node.js Runtime で動作させる場合の違いを理解した実装か確認する。

チェック観点：
- `export const config = { runtime: 'edge' }` を設定している場合、Node.js 専用 API（`fs`・`process.env` 以外のモジュール等）を使っていないか
- Edge Runtime では `crypto.randomUUID()` 等の Web 標準 API が使えるが、Node.js 固有 API は使えないことを把握しているか
- このプロジェクトの既存 API（`episodes.js`・`episode-meta.js` 等）は Node.js Runtime を前提としているため、新しい API を Edge Runtime にする場合は明示的な理由が必要か
- Edge Runtime を使う場合、`@upstash/redis` のクライアントが Edge 対応版（`@upstash/redis` は対応済み）になっているか

## 5. API RateLimit 考慮

外部 API（YouTube Data API・Google Sheets API）の利用量が増加する変更がないか確認する。

チェック観点：
- YouTube Data API の1日あたりのクォータ（デフォルト 10,000 ユニット）を超える呼び出しが発生しないか
- キャッシュ（Redis）を通さずに YouTube API・Sheets API を直接呼ぶコードパスが追加されていないか
- `?refresh=1` などの強制更新を誰でも叩ける状態になっていないか（認証が必要）
- エラー時のリトライロジックが指数バックオフなしでループしていないか
- フロントエンドから直接外部 API を呼ぶ実装（CORS・APIキー露出リスク）が追加されていないか

## 6. JSON キャッシュ戦略

Redis キャッシュと `data/*.json` フォールバックの戦略が維持されているか確認する。

チェック観点：
- Redis から取得 → 失敗時は `data/*.json` フォールバック、の二段構えになっているか
- 新しいキャッシュキーを追加する場合、TTL が設定されているか（無期限キャッシュは NG）
- YouTube API データの TTL が 30 日以内か（YouTube API ToS）
- Google Sheets メタ情報の TTL が適切か（現在: 3600 秒。長くする場合は理由が必要）
- キャッシュキーのバージョニング（`_v1` サフィックス等）が既存パターンに合っているか
- `data/episodes.json` / `data/episodeMeta.json` のフォールバックデータが最新の状態を保っているか

## 7. 差分同期優先

全件取得・全件上書きではなく、差分のみを更新する設計になっているか確認する。

チェック観点：
- YouTube Playlist の全件取得が毎回発生していないか（変更のないエピソードは再取得不要）
- Redis への書き込みが全件上書きになっていないか（差分があった分だけ更新すること）
- Google Sheets からの取得が毎リクエスト発生していないか（Redis キャッシュを挟むこと）
- 新しい同期処理を追加する場合、「前回取得した状態」をどこかに保持して差分判定しているか
- Cron による定期更新と手動 `workflow_dispatch` による即時更新が、重複して全件書き込みを起こさないか

---

チェック完了後、「Vercel 運用方針に沿っている」「修正してからデプロイ可」「設計から見直しが必要」のいずれかを判定し、修正が必要な場合は優先度順に提案を示してください。

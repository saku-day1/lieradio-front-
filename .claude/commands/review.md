まず `git diff main...HEAD` でブランチの変更差分を取得し、以下の7項目を順番にチェックして結果を報告してください。

各項目は「✅ 問題なし」「⚠️ 要確認」「❌ アウト」の3段階で評価し、問題がある場合は該当箇所（ファイル名・行番号）と理由を具体的に示すこと。

---

## 1. 責務分離確認

変更されたファイルが CLAUDE.md の責務分離ルール（各モジュールの責務表）に違反していないか確認する。

チェック観点：
- `AppController.js` が DOM 直接描画・データ取得・localStorage 操作をしていないか
- `EpisodeListView.js` / `RankingView.js` が状態の永続化・フィルタロジックを持っていないか
- `EpisodeRepository.js` が DOM 操作・localStorage 操作をしていないか
- `UserPreferences.js` が DOM 操作・データフィルタリングをしていないか
- View → Controller の通知がコールバック関数を経由しているか（直接参照していないか）

## 2. 不要な依存追加確認

新しいライブラリ・外部モジュール・npm パッケージが追加されていないか確認する。

チェック観点：
- `package.json` に新しい `dependencies` / `devDependencies` が追加されていないか
- `import` / `require` で新しい外部モジュールを参照していないか
- CDN リンクが `index.html` に追加されていないか
- 追加されている場合、既存の仕組みで代替できないか判断する

## 3. main 影響確認

変更が既存の主要機能（フィルタ・ページネーション・エピソード表示・お気に入り・視聴済み）に予期せぬ影響を与えていないか確認する。

チェック観点：
- `AppController.js` のフィルタ状態管理ロジックに破壊的変更がないか
- `EpisodeRepository.js` のフィルタ・ソート・集計ロジックに変更がある場合、既存フィルタが正しく動作するか
- `UserPreferences.js` の localStorage キー名が変更されていないか（変更するとお気に入り・視聴済みデータが消える）
- `data/episodeMeta.json` のスキーマ変更がフロントのパースロジックと整合しているか

## 4. 法的リスク確認

著作権・個人情報・API 利用規約に抵触する変更がないか確認する。

チェック観点：
- 動画・音声・画像ファイルが `data/` 等にコミットされていないか
- 番組・放送の内容テキスト（台本・セリフ等）が転載されていないか（タイトル・URLリンクは可）
- YouTube API のレスポンスをキャッシュする場合、TTL が 30 日以内か
- サーバー側でユーザーの個人情報（メール・位置情報等）を新たに収集・保存していないか
- APIキー・トークン・パスワードがコードや `data/` にハードコードされていないか

## 5. API コスト確認

YouTube API・Google Sheets API・Upstash Redis の呼び出し回数・コストが増加する変更がないか確認する。

チェック観点：
- YouTube Data API の呼び出しが増える変更（ページネーションなし全件取得・キャッシュバイパス等）がないか
- Google Sheets API の呼び出しが毎リクエスト発生するような実装になっていないか（Redis キャッシュを通すこと）
- Redis への書き込みが全件上書きになっていないか（差分更新を優先）
- Cron の実行頻度が増えていないか（`vercel.json` の schedule 変更）

## 6. 命名整合性確認

新しく追加した変数名・関数名・JSON フィールド名・キャッシュキーが既存の命名規則と整合しているか確認する。

チェック観点：
- JS 変数・関数名が camelCase で既存コードと統一されているか
- `episodeMeta.json` の新フィールドが既存フィールド（`corners`, `tags`, `lunchTimeRequestSong` 等）と命名スタイルが揃っているか
- Redis キャッシュキーが `episode_meta_cache_v1` のような既存パターンに従っているか
- 新しい概念・抽象（レイヤー・クラス名等）を不必要に増やしていないか

## 7. キャッシュ破壊確認

Redis キャッシュや `data/*.json` の既存データを破壊・無効化する変更がないか確認する。

チェック観点：
- Redis キャッシュキー名を変更していないか（変更するとキャッシュが即座に失効する）
- `data/episodes.json` / `data/episodeMeta.json` を意図せず上書きするスクリプト変更がないか
- `invalidateCache` / `refresh` エンドポイントの認証チェックが維持されているか
- TTL を短縮する変更がないか（API コスト増につながる）
- キャッシュなしで直接 API を叩くフォールバックを追加した場合、レートリミットへの影響はないか

---

チェック完了後、問題がなければ「マージ可」、要確認・アウト項目がある場合は「マージ前に対応が必要」と最後に明記してください。

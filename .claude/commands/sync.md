データ同期処理（YouTube API 取得・Google Sheets 取得・Redis キャッシュ更新・`data/*.json` 更新）の新規追加または既存変更を対象に、以下の5項目を順番にチェックしてください。

各項目は「✅ 問題なし」「⚠️ 要注意」「❌ 設計違反」の3段階で評価し、問題がある場合は該当箇所と改善案を具体的に示すこと。

---

## 1. fingerprint 生成

データの変化を検知するための識別子が適切に生成されているか確認する。

チェック観点：
- 同期対象のデータに「いつ・何が変わったか」を判定できるハッシュ・ETag・バージョンフィールドがあるか
- YouTube Playlist の更新検知に `publishedAt` / `updatedAt` / `etag` を活用しているか
- Google Sheets の変更検知に行数・最終更新日・スプレッドシートの `modifiedTime` を使えるか
- fingerprint の生成に外部依存（UUID ライブラリ等）を追加していないか（`crypto.createHash` / `crypto.randomUUID` で代替できるか）
- Redis に保存する fingerprint のキー名が既存パターン（`_v1` サフィックス等）に合っているか

## 2. 差分検知

前回の同期結果と比較して、変化があった部分だけを処理しているか確認する。

チェック観点：
- 全件取得 → 全件上書きになっていないか（前回の fingerprint と比較して差分があった場合のみ書き込むこと）
- `videoId` を主キーとして差分を判定しているか（タイトル・公開日・配列順で比較していないか）
- 新規追加・更新・削除を区別して処理しているか
- 差分がない場合は Redis 書き込みをスキップしているか（不必要な TTL リセットを防ぐ）
- 差分検知のロジックが `videoId` 完全一致を前提としているか（CLAUDE.md: データ結合ポリシー）

## 3. 更新日時比較

「最後に同期した日時」と「ソースデータの更新日時」を比較して、同期の必要性を判断しているか確認する。

チェック観点：
- Redis に「最終同期日時」または「ソースの最終更新日時」を保存しているか
- 更新日時の比較に UTC を基準としているか（タイムゾーン混在による誤検知を防ぐ）
- YouTube API の `snippet.publishedAt` が ISO 8601 形式であることを前提に正しくパースしているか
- Google Sheets の `modifiedTime`（Drive API）を使う場合、取得コストと効果を検討しているか
- Cron による定期実行と手動 `workflow_dispatch` による即時実行で、更新日時の比較ロジックが共通化されているか

## 4. fail-safe

同期処理が失敗しても、サイトの表示・機能が継続できるか確認する。

チェック観点：
- YouTube API・Sheets API・Redis のいずれかが失敗しても、`data/episodes.json` / `data/episodeMeta.json` のフォールバックが機能するか
- 同期処理の途中で失敗した場合、中途半端な状態（一部だけ更新された Redis）にならないか
- エラー時に前回の正常なキャッシュが残っているか（TTL が切れていないか）
- 同期失敗を検知して通知・ログ出力する仕組みがあるか（`console.error` 最低限・GitHub Actions ログ）
- 失敗した場合の `try/catch` が適切に書かれており、エラーを握りつぶしていないか

## 5. rollback

同期処理が誤ったデータを書き込んだ場合に、元の状態に戻せるか確認する。

チェック観点：
- Redis に書き込む前に、現在のキャッシュ内容をバックアップ（別キー保存）しているか、または TTL 内であれば旧データが取れる設計になっているか
- `data/episodes.json` / `data/episodeMeta.json` を上書きする処理の場合、git 履歴から前バージョンに戻せるか（コミットされているか）
- キャッシュキーのバージョンを上げる（`_v1` → `_v2`）変更をする場合、旧キーを即時削除せずに TTL で自然消滅させているか
- 誤ったデータで Redis を上書きしてしまった場合の手順（`?refresh=1` で再取得・`data/*.json` で手動フォールバック）がドキュメント化されているか

## 6. 冪等性

同じ同期処理を複数回実行しても、結果が同じになるか確認する。

チェック観点：
- Cron が重複して実行された場合（前回の実行が終わる前に次が起動した場合）に、データが二重書き込みや不整合にならないか
- `?refresh=1` を連打した場合に、API クォータを大量消費したりデータが壊れたりしないか
- 同じ `videoId` のデータを2回処理した場合、重複エントリが生成されないか（upsert 相当の処理になっているか）
- 初回実行とそれ以降の実行で、異なる副作用（ファイル生成・キー作成等）が起きないか
- `workflow_dispatch` による手動実行と Cron 自動実行が同時に走っても問題ないか

---

チェック完了後、「同期設計に問題なし」「部分修正が必要」「同期設計の見直しが必要」のいずれかを判定し、修正が必要な場合は優先度順に提案を示してください。

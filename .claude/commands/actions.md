`.github/workflows/` 以下の変更、または会話中で示された GitHub Actions の追加・修正を対象に、以下の6項目を順番にチェックしてください。

各項目は「✅ 問題なし」「⚠️ 要注意」「❌ 違反」の3段階で評価し、問題がある場合は該当ファイル・行と修正案を具体的に示すこと。

---

## 1. secrets 直書き禁止

ワークフロー内に認証情報・トークン・APIキーがハードコードされていないか確認する。

チェック観点：
- `env:` や `run:` ブロックに APIキー・トークン・パスワードが文字列で書かれていないか
- `${{ secrets.XXX }}` を使わずに値を直接渡していないか
- ログ出力（`echo`・`cat` 等）で secrets の値が露出しないか（`echo ${{ secrets.FOO }}` は NG）
- `curl` コマンドのヘッダーや URL クエリに secrets が直接埋め込まれていないか

このプロジェクトで使用する secrets：`YOUTUBE_API_KEY` / `CRON_SECRET` / `GOOGLE_SHEETS_API_KEY` / `KV_REST_API_URL` / `KV_REST_API_TOKEN` など。新しい secrets を追加した場合、Vercel の Production・Preview・Development 全スコープへの設定も確認すること。

## 2. 本番 DB・外部サービス接続禁止

CI 実行中に本番の外部サービスへ書き込むワークフローになっていないか確認する。

チェック観点：
- テスト・検証ステップが本番 Upstash Redis（`KV_REST_API_URL`）に書き込んでいないか
- `?refresh=1` などのキャッシュ強制更新エンドポイントをテスト中に呼んでいないか
- 本番 YouTube API や Google Sheets API に対して不必要な書き込み・更新リクエストを送っていないか
- `workflow_dispatch` 以外のトリガー（`push`・`pull_request` 等）で本番データを変更するステップが含まれていないか

本番への書き込みが必要なステップは `workflow_dispatch` に限定し、テスト用途では Upstash の別環境キーまたはモックを使うこと。

## 3. インメモリ・モック利用（本番外サービス代替）

CI の自動テストでデータストアが必要な場合、本番サービスの代わりにインメモリ・モックを使っているか確認する。

チェック観点：
- Java プロジェクト等で DB が必要な場合、H2 などのインメモリ DB を使っているか（本番 DB には接続しない）
- このプロジェクト（Redis）では、統合テストに本番 Upstash ではなく `redis-mock` や Upstash のテスト環境を使っているか
- `api/*.js` の単体テストが本番エンドポイントに依存していないか
- モックで代替できない場合、その理由がコメントに明記されているか

## 4. 差分 deploy

毎回フルビルド・フルデプロイを行う設計になっていないか確認する。

チェック観点：
- 変更のないファイルまで毎回デプロイしていないか
- `paths:` フィルタや `on: push: paths:` を使って変更があったファイルにのみ実行するようにしているか
- キャッシュ更新 workflow（`refresh-meta.yml` 等）が `push` ごとに不必要にトリガーされていないか
- Vercel の自動デプロイと workflow のデプロイが二重になっていないか

このプロジェクトのデプロイは Vercel 自動デプロイが担うため、workflow は「キャッシュ更新」「メタ情報反映」に限定すること。

## 5. action version 固定

使用する GitHub Actions のバージョンが固定されているか確認する。

チェック観点：
- `uses: actions/checkout@v4` のように **メジャーバージョン以上**で固定されているか
- `@main` / `@master` / `@latest` などの浮動参照を使っていないか
- サードパーティ action（`actions/` 以外）は **コミット SHA** で固定されているか（例: `uses: foo/bar@abc1234`）
- 既存 workflow と異なるバージョンを使う場合、理由が明記されているか

このプロジェクトで使用中のバージョン：`actions/checkout@v4`。新しい action を追加する場合は同様のポリシーに従うこと。

## 6. rollback 考慮

デプロイ・キャッシュ更新が失敗したときに、元の状態に戻せる設計になっているか確認する。

チェック観点：
- キャッシュを上書きするステップが失敗した場合、前回のキャッシュが生きているか（TTL 切れにならないか）
- `episodes-refresh` や `refresh-meta` の失敗時に `data/episodes.json` / `data/episodeMeta.json` のフォールバックが機能するか
- workflow 失敗時に Slack 通知・ログ出力・`exit 1` など失敗を検知できる仕組みがあるか
- 新しいキャッシュキーに切り替える場合、旧キーのデータを即時削除していないか（段階移行）
- `continue-on-error: true` を使っている場合、失敗を隠蔽していないか

---

チェック完了後、「デプロイ可」「修正してから実行」「設計見直しが必要」のいずれかを判定し、次のアクションを1〜3行で提示してください。

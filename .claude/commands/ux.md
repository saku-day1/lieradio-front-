提案中または実装済みの UI 変更・新機能を対象に、以下の5項目を順番にチェックしてください。

各項目は「✅ 問題なし」「⚠️ 要検討」「❌ 方針違反」の3段階で評価し、問題がある場合は具体的な改善案を示すこと。

---

## 1. ライト層優先

このサイトの主なユーザーは「リエラジを聴いたことがある・気になっている」程度のカジュアルなリスナーを想定する。コア視聴者だけが使える機能になっていないか確認する。

チェック観点：
- 操作に前提知識（キャスト名・コーナー名・放送回数）が必要になっていないか
- 初めて訪れたユーザーが何をするサイトか3秒で理解できるか
- フィルタ・検索の初期状態が「何も選んでいない＝全件表示」になっているか（ライト層はフィルタしない）
- エラーや空状態のメッセージがサイト内文脈を知らなくても意味が通じるか
- 新しい機能がライト層を混乱させる複雑さを追加していないか

## 2. 1クリックで目的到達

ユーザーが「特定のキャストの回を見たい」「お気に入りを確認したい」などの主要な目的を、最小限の操作で達成できるか確認する。

チェック観点：
- クイックフィルタボタン（キャスト・ユニット）がファーストビューに見えているか
- お気に入り・視聴済みへのアクセスが1タップで届くか
- 最も使われる操作（キャスト単体フィルタ）がスクロールなしで操作できるか
- 新しい機能を追加した場合、それにたどり着くまでのタップ数は何回か（3回以上なら再検討）
- モーダル・パネル・アコーディオンを増やした場合、「閉じる」操作が直感的か

## 3. 情報量を増やしすぎない

エピソードカード・フィルタ・詳細パネルに表示する情報が過剰になっていないか確認する。

チェック観点：
- エピソードカードに新しい情報を追加した場合、既存の情報（タイトル・出演者・日付）が埋もれていないか
- 1画面に表示される要素の種類が増えていないか（ボタン・バッジ・アイコン・テキストの密度）
- リクエスト曲（`lunchTimeRequestSong`）は「詳細」を開いた中にのみ表示しているか（CLAUDE.md 仕様）
- 詳細パネルを開かないと基本情報が得られない設計になっていないか（逆方向の情報隠蔽）
- 追加した表示要素を「出さなかった場合」と比較して、本当に必要か再評価する

## 4. コア機能を埋もれさせない

新しい機能・情報の追加によって、既存のコア機能（キャストフィルタ・エピソードリスト・お気に入り）が見つけにくくなっていないか確認する。

チェック観点：
- キャスト別クイックフィルタボタンがページ上部に見えているか
- 新しいフィルタ・ボタン・セクションを追加した場合、クイックフィルタの視認性が下がっていないか
- エピソードリストが画面の主要領域を占めているか（ランキング・バナー等に圧迫されていないか）
- 詳細検索パネルが「展開しないと基本フィルタが使えない」構造になっていないか
- 追加した UI 要素の優先度が、既存コア機能より低いことを視覚的に表現できているか

## 5. モバイル優先

スマートフォン（縦向き、幅 375px 想定）で快適に操作できるか確認する。

チェック観点（CLAUDE.md: スマートフォン向けユーザーを主対象とする）：
- タップターゲットが十分な大きさか（最低 44×44px）
- フォントサイズが読みやすいか（本文 14px 以上、補足 12px 以上）
- 横スクロールが発生していないか
- ボタン・フィルタ・カードが縦積みレイアウトで崩れていないか
- 新しく追加したインタラクション（スワイプ・長押し等）がモバイルで自然に動作するか
- PC での表示が破綻していないか（モバイル優先だが PC で壊れていい理由はない）

---

チェック完了後、「UX 方針に沿っている」「一部改善してリリース可」「設計から見直しが必要」のいずれかを判定し、改善が必要な場合は優先度順に提案を示してください。

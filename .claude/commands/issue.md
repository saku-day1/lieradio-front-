現在のブランチの変更差分（`git diff main...HEAD`）または会話中で示された実装タスクを対象に、以下の5項目を順番に分析し、Issue として切り出すべき単位を提案してください。

ユーザーが実装タスクを文章で説明している場合はその内容を、差分がある場合は差分を分析対象とします。

---

## 1. スコープ分割

「1つのPR・1つの会話」に含まれる変更を、独立してレビュー・マージ・ロールバックできる単位に分解する。

分析観点：
- UI変更・ロジック変更・データ変更・インフラ変更が混在していないか
- 「〜も直した」「ついでに〜」という追加変更が含まれていないか
- 変更ファイル数が多い場合、責務ごとに分割できないか（例: View だけ / Controller だけ / API だけ）
- 分割した場合、それぞれが単独でリリースできるか（依存関係を確認）

出力：分割案をリスト形式で提示する。分割不要なら「分割不要」と明記する。

## 2. 工数肥大化検知

当初の目的に対して実装範囲が膨らんでいないか確認する。

分析観点：
- 変更行数・変更ファイル数が目的に対して過大でないか
- 本来不要なリファクタ・抽象化・汎化が混入していないか（CLAUDE.md: 不要な抽象化をしない）
- 「将来のため」「汎用的にするため」の実装が含まれていないか
- 新しいレイヤー・クラス・概念が追加されている場合、既存の仕組みで代替できないか

出力：肥大化している箇所とその理由を具体的に指摘する。問題なければ「スコープ適切」と明記する。

## 3. MVP 優先

今リリースすべき最小限の変更（MVP）と、後回しにできる変更を分類する。

分析観点：
- ユーザーが今すぐ困っている問題を解決する変更はどれか
- 「あると便利」「polish」レベルの変更はどれか
- 今リリースしないと次の作業がブロックされる変更はどれか
- MVP だけでも動作として成立するか

出力：
- **今すぐリリース（MVP）**: ～
- **次のイテレーション**: ～
- **バックログ（優先度低）**: ～

## 4. 依存関係整理

分割した Issue 間の実装順序・依存関係を整理する。

分析観点：
- A を先にリリースしないと B が動かない、という依存があるか
- データ構造変更（`episodeMeta.json` スキーマ・Redis キャッシュキー等）が先行して必要か
- フロント変更と API 変更を同時にリリースしなければならない箇所があるか
- 並行して実装できる独立した Issue があるか

出力：依存グラフまたは実装順序を箇条書きで示す。

## 5. 後回し判断

今回のスコープから外して Issue に積むべき事項を判断する。

判断基準：
- 今回の変更目的と直接関係しない改善・修正
- 影響範囲が広く、単独で設計検討が必要なもの
- 現時点で仕様が未確定のもの（CLAUDE.md: 仕様が曖昧な場合は確認を優先）
- 「いつかやりたい」レベルで今すぐマージを阻むべきでないもの

出力：後回し事項を Issue タイトル案つきでリスト化する。形式：`[ ] <タイトル> — <理由>`

---

最後に、今回のスコープを一言でまとめ、「このままマージ可」「スコープを絞ってからマージ推奨」「設計検討が必要」のいずれかを判定してください。
